# 🧠 Agent Memory Architecture

## 大脑 + 书架：给 AI Agent 一个真正的长期记忆系统

> **你的 AI Agent 记不住东西，不是因为它不行，是因为你没有给它一个像样的记忆系统。**

大多数 AI Agent 的记忆要么是"塞进上下文 prompt 里随用随丢"，要么是"扔进向量数据库却不知道怎么查回来"。这套方案是第三条路——**零依赖、纯文件、grep 可搜**。

---

## 核心理念

```
┌─────────────────────────────────────┐
│     MEMORY.md / USER.md              │  ← 桌面笔记本（会话注入，常翻常新）
│     (3-5K chars, session-injected)   │     精简到 3-5K chars
├─────────────────────────────────────┤
│          ~/details/                   │  ← 图书馆索引台（结构化 markdown）
│     (按主题分类, read_file)           │     完整记录，随时查阅
├─────────────────────────────────────┤
│       ~/details/archive/             │  ← 书房书架（全文存档）
│    (grep 全文检索, 附时间戳索引)      │     永不丢失，永不压缩
└─────────────────────────────────────┘

    ↓ 每周六凌晨自动归档（14天无活动事件）→ 像大脑 NREM 睡眠的清洗周期
```

### 为什么你不需要向量数据库

| 方案 | 依赖 | 运维 | 可检索 | 成本 |
|------|------|------|--------|------|
| 向量数据库 (mem0, MemGPT) | 外部服务 + 嵌入模型 | ❌ 重 | ✅ 语义搜索 | ❌ 高 |
| 全量塞 prompt | 无 | ✅ 轻 | ❌ 无结构 | ❌ 上下文爆炸 |
| **本方案（大脑+书架）** | **❌ 零依赖** | **✅ 纯文件** | **✅ grep/rg** | **✅ 免费** |

文件系统就是你的数据库。`grep` 就是你的语义引擎。cronjob 就是你的睡眠清洗周期。

## 科学原理

这套架构不是拍脑袋想出来的——它直接映射了人脑记忆系统的三层结构，以及人类文明几千年来的知识传承方式。

### 人脑记忆的三个层次

人脑的记忆不是铁板一块，而是分层的：

```
感官记忆（~秒）   →   工作记忆（~分钟）   →   长期记忆（数年～终身）
   ↓                      ↓                        ↓
  大量原始信号           当前关注的7±2项            索引化的知识网络
  持续覆盖               用完即丢                  睡眠中巩固
```

**关键洞察**：人脑的"短期"和"长期"不是容量大小的区别，是**索引方式**的区别。

工作记忆里的信息是通过"当前活跃的神经回路"直接访问的——快、但没有结构。长期记忆是通过**索引**访问的——海马体像图书馆的索引卡一样，告诉你去新皮层的哪个位置找那段记忆。

### 睡眠——短期变长期的关键

每天产生的大量短期记忆，在睡眠中被分类、筛选、巩固到长期存储：

1. **NREM 睡眠**：去甲肾上腺素每 ~50 秒振荡一次，驱动脑脊液冲刷代谢废物（2025 Cell 论文）。同时海马体反复回放白天的经历，把重要信息"写入"新皮层。
2. **REM 睡眠**：在不同记忆片段之间建立新的连接——这就是为什么睡一觉后能想到之前想不到的关联。

**没有睡眠，短期记忆就是一堆碎片，塞车在脑子里**。

### 人脑记忆的局限性

人脑的强大在于联想、抽象、模式提取。但它有三个硬伤：

| 缺陷 | 表现 | 后果 |
|------|------|------|
| **容量有限** | 工作记忆 7±2 项，长期记忆会模糊/扭曲 | 细节随时间丢失 |
| **不可靠** | 每次回想都会重建而非回放，容易掺假 | 记忆会被改写 |
| **检索不稳定** | 靠联想触发，不保证能找到 | 想不起来就是不存在 |

### 为什么人类发明了文字和书

文字是**外部的、稳定的、精确的**记忆系统。书是最好的例子：

```
                    大脑                                书
         ┌──────────────────────┐        ┌──────────────────────┐
         │  快速联想             │        │  精确记录            │
         │  模式提取             │        │  容量无限            │
         │  抽象理解             │        │  永不扭曲            │
         │  但容量有限           │        │  但检索慢            │
         │  会扭曲/遗忘          │        │  需要索引            │
         └──────────────────────┘        └──────────────────────┘
                      ↕ 配合使用 ↕
              大脑提供方向和判断
              书提供精确的细节
```

人类从来不是"只靠大脑记忆"的物种——我们是**大脑 + 外脑**协同的物种。书、笔记、数据库，都是外脑。这套架构只是把这个几千年验证过的模式搬到了 AI Agent 身上。

### 这套架构 = 大脑 + 书架

| 人脑/人类文明 | 我们的系统 | 说明 |
|-------------|-----------|------|
| 工作记忆（当前关注的几件事） | **MEMORY.md / USER.md** | 会话注入，随取随用，精简到 3-5K chars |
| 海马体索引 | **MEMORY.md 中的索引行** | `项目名 -> details/项目名.md`，指向详细记录 |
| 新皮层长期存储 | **~/details/ 主目录** | 结构化分类存储，按主题归档 |
| 睡眠巩固（短期→长期） | **cronjob 归档** | 每周六凌晨，识别过时事件搬到 archive |
| 睡眠清洗（去甲肾上腺素振荡） | **低频脉冲式整理** | 不做实时监控，把整理和工作分离 |
| 写书/做笔记（外脑存档） | **~/details/archive/** | 全文保存，永不删除，grep 可搜 |
| 书的索引/目录 | **archive/README.md** | 每次归档自动追加索引 + 关键词 |
| 去图书馆查书 | **grep -r "关键词" ~/details/** | 全文检索，一秒定位 |

### 这样设计的好处

1. **工作时不整理，整理时不工作**——和睡眠一样，清洗和操作分离
2. **"遗忘"不是删除，是移到书架**——和出书一样，知识进图书馆而不是进垃圾桶
3. **索引在脑，内容在书**——MEMORY.md 只有一行 `项目名 -> details/xxx.md`，就像你记得某本书在图书馆的哪一排，不记得每一页的内容
4. **定期的、脉冲式的整理**——不是每次对话后都整理（那太碎了），也不是永远不整理（那太乱了），每周一次刚刚好

### 模型无关

这套架构就是文件操作 + cronjob，任何模型都能用。

## 一条命令安装

```bash
curl -fsSL https://raw.githubusercontent.com/caoyiqing1987/agent-memory-architecture/main/install.sh | bash
```

## 快速开始

```bash
# 1. 编辑你的记忆文件
vim ~/.hermes/memories/MEMORY.md   # 系统规则 + 索引
vim ~/.hermes/memories/USER.md     # 你的偏好 + 背景

# 2. 添加完整记录到 details/
mkdir -p ~/details/projects
cat > ~/details/projects/my-project.md << 'EOF'
# 我的项目
- 客户：xxx
- 状态：进行中
- 交期：6月
EOF

# 3. 在 MEMORY.md 里加一行索引
#    my-project -> details/projects/my-project.md

# 4. 归档脚本每周六自动跑，或者手动：
python3 ~/.hermes/scripts/archive-memory.py
```

## 三层架构详解

### 第一层：MEMORY.md + USER.md（桌面笔记本）

随每次会话注入 system prompt，是你 AI Agent 的"工作记忆"。

**应该放什么：**
- 系统规则（工具偏好、行为约束、输出格式）
- 用户偏好（沟通风格、决策习惯、重要背景）
- 指向 details/ 的短索引行：`项目名 -> details/projects/项目名.md`

**不应该放什么：**
- ❌ 事件完整记录
- ❌ 过时状态
- ❌ 大段描述性文字

### 第二层：~/details/（图书馆索引台）

结构化分类存储，按主题放 markdown 文件。`read_file` 随时读。

推荐目录结构：
```
details/
├── projects.md          # 项目状态汇总
├── business.md          # 业务/客户信息
├── team.md              # 团队分工
├── personal.md          # 个人/家庭
├── archive/             # 归档目录
│   ├── README.md        # 搜索索引
│   ├── memory-archive.md
│   └── user-archive.md
```

### 第三层：~/details/archive/（书房书架）

全文存档，永不删除。每条归档附带：
- 时间戳（哪一天归档的）
- 主题标签（财务/项目/技术/医疗/法务/生活）
- 搜索关键词（便于 grep）

**检索：**
```bash
grep -r "关键词" ~/details/          # 查活跃记录
grep -r "关键词" ~/details/archive/  # 查归档
```

---

## 归档机制（睡眠模型）

- **频率**：每周一次（周六凌晨 4:00），低频脉冲
- **条件**：14 天前的单次事件（带日期的明显过时内容）
- **保护**：系统规则、索引行、偏好信息永不归档
- **输出**：原文完整保存到 archive + 索引追加到 README.md

```bash
# 手动触发归档
python3 ~/.hermes/scripts/archive-memory.py
```

---

## 配置文件

在 `~/.hermes/config.yaml` 中：
```yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 50000   # MEMORY.md 上限
  user_char_limit: 50000     # USER.md 上限
```
